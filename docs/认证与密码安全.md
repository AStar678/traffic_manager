# 认证与密码安全设计

## 实现目标

本实现对应任务清单“用户隐私数据在传输和存储阶段采用加密手段”要求：

- 登录、验证码登录、注册和验证码发送请求不传输明文账号信息。
- 数据库不存储可还原的用户密码。
- 密钥、数据库凭据和 JWT 密钥不进入仓库。

## 传输加密

1. 前端请求 `GET /api/v1/auth/encryption-key` 获取 RSA 公钥和指纹。
2. 每次认证请求生成独立的 256 位 AES 密钥与 96 位随机 IV。
3. 请求 JSON 用 AES-256-GCM 加密，公钥指纹作为 AAD，同时生成 128 位认证标签以检测篡改。
4. AES 密钥用 RSA-3072 OAEP-SHA256 加密。
5. 后端校验指纹、长度与 GCM 标签，解密后再执行 Jakarta Validation。旧的明文请求格式会被直接拒绝。

前端在 HTTPS/安全上下文中优先使用 Web Crypto API；当实训环境使用普通 HTTP 私网 IP 时，使用锁定版本的 `node-forge` 完成相同算法。应用层加密用于避免明文出现在请求体中，正式公网部署仍应配置 HTTPS，以完成服务器身份认证和保护所有响应数据。

## 密码存储与迁移

- 新注册密码使用 BCrypt cost 12 的随机盐哈希。
- 后端启动时在事务中扫描历史用户，将非 BCrypt 字段一次性转换并立即刷入数据库。
- 迁移后仍存在非 BCrypt 字段时，服务拒绝完成启动。
- 登录逻辑只接受符合 BCrypt 格式的数据库值，不再兼容明文比对；旧 cost 参数会在成功登录时升级。

运维审计只需统计不合规记录数，不应输出完整哈希，更不应输出明文密码。

## 密钥运维

服务器首次部署会生成 `/home/aoxiang/visiondrive/config/auth-private-key.pem`，权限为 `600`。该文件不会随代码同步，重启后继续使用同一密钥。本地开发未配置私钥路径时仅使用进程级临时密钥。
